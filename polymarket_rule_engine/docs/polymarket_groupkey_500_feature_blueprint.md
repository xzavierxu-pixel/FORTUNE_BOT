# Polymarket group_key 分层特征蓝图（500 个特征）

## 1. 我对你当前 group_key 的理解

我读取了你上传的 `trading_rules.csv`。当前 `group_key` 的结构是：

```text
group_key = domain | category | market_type
```

- 唯一 `group_key` 数：**152**
- 唯一 `domain` 数：**105**
- 唯一 `category` 数：**7**
- 唯一 `market_type` 数：**5**

这说明你的分组天然就是一个**三层可拆分的层级键**，非常适合做：

1. 最细粒度：`full_group_key = domain|category|market_type`
2. 中间层：`domain`、`category`、`market_type`
3. 两两组合层：`domain×category`、`domain×market_type`、`category×market_type`
4. 全局层：`global`

也就是说，你现在的 group_key **不是只能做一个组统计**，而是可以自然扩展成一整套分层历史特征。

- `liquipedia.net/counterstrike|SPORTS|other`
- `twitch.tv|SPORTS|other`
- `ncaa.com.football|SPORTS|over_under`
- `ncaa.com.football|SPORTS|other`
- `ncaa.com.basketball|SPORTS|other`
- `UNKNOWN|SPORTS|other`
- `binance.com/en/trade/XRP_USDT|CRYPTO|no_yes`
- `pgatour.com|SPORTS|no_yes`

### 1.2 当前文件里能直接利用的规则表字段

你上传的 `trading_rules.csv` 里除 `group_key/domain/category/market_type` 外，还包含：

- `leaf_id`

这些字段很适合做**规则先验特征**，我已经单独给你放进 32 个 feature。

## 2. 设计原则

### 2.1 粗筛和模型输入分开

- **粗筛**：你已经决定用 `Brier + logloss`，这是合理的。
- **模型输入**：可以把更多历史行为特征、尾部特征、规则先验、交互项一起喂给 LightGBM。

### 2.2 所有历史统计必须严格防 leakage

下文所有 `recent_50 / recent_200 / expanding` 统计，都必须满足：

- 只使用**当前样本预测时点之前**已经 resolved 的市场
- 最好按你实际建模的 horizon 版本分别计算
- 如果你做 pooled model，至少要保留 `当前 horizon` 对应版本

### 2.3 先做层级统计，再让树模型自己选

不要只做最细 `full_group_key`。推荐同时保留：

- `global`
- `domain`
- `category`
- `market_type`
- `domain×category`
- `domain×market_type`
- `category×market_type`
- `full_group_key`

这样样本不足时，模型还能从更粗层级借信息。
